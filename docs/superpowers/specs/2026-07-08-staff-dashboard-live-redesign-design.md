# Staff Dashboard Live Redesign — Design

**Context anchor.** Epic: Digital Ordering Core Loop · Bounded context: Ordering (staff-facing + customer-facing) · Builds on Story 7 (`docs/superpowers/specs/2026-07-04-story-7-staff-dashboard-pending-orders-design.md`), Story 8 (confirm/pay), and the confirmation-page redesign (`2026-07-07-order-confirmation-page-redesign-design.md`). Follows ADR-001 (`04-architecture.md`) for the polling rationale. Post-epic, user-directed (like Story 9).

**Motivation.** The staff dashboard currently shows one flat, static list of Pending orders with inline action buttons and no prices — confirming an order just makes it vanish, with no record visible on-screen. The customer confirmation page (`/order/[id]`) only discovers a staff confirmation when the customer manually reloads or attempts an edit and hits a 409. This pass makes the dashboard feel responsive and alive (entrance/exit animation, a real interaction pattern), adds a same-day record of completed orders instead of orders disappearing outright, and closes the live-update gap on the customer side so the lock takes effect the moment staff acts, not the next time the customer happens to reload.

**Scope.** Touches `app/dashboard/PendingOrdersDashboard.tsx`, `app/dashboard/page.tsx`, `app/globals.css`, `lib/orderService.ts` (`listOrders` filters only), `app/api/orders/route.ts` (new query params), `app/order/[id]/page.tsx`, `app/order/[id]/OrderTicket.tsx`, `app/order/[id]/ConfirmDialog.tsx`, `app/order/OrderReviewModal.tsx`. New files: `app/components/Modal.tsx` (shared modal chrome), `app/dashboard/OrderDetailModal.tsx`, `app/api/orders/[id]/status/route.ts`, `app/order/[id]/OrderStatusPoller.tsx`. No changes to `POST/PATCH/DELETE` mutation routes, `confirmOrder`/`setPaymentStatus`/`cancelOrder`/`removeOrderItem` logic, or any state-machine invariant — `INV-4/5/8/9` are enforced exactly as today; this pass only changes *when* the UI reflects them.

## Decisions

### 1. Three-lane dashboard instead of one flat list

- **Problem.** Today's dashboard fetches only `status=pending` and renders one list. Confirming an order removes it from the screen entirely — there's no on-screen trace that it happened, and no way to find a `Confirmed`-but-`Unpaid` order to mark it paid later (a legal transition per `INV-8`: payment can be marked independently of, and after, confirmation).
- **Fix.** Three visual groups:
  1. **Pending** — unchanged data source (`GET /api/orders?status=pending`), redesigned cards (Decision 2).
  2. **Confirmed & Unpaid** — new lane. Orders that are `Confirmed` but still `Unpaid` stay as full cards here, since they still need a staff action. Not date-scoped — an unpaid confirmed order from an earlier session should never silently disappear from the actionable view.
  3. **Completed today** — a small counter chip (e.g. "14 completed today"), covering orders that are `Confirmed` **and** `Paid`, scoped to the current calendar day. This is the "historical record" the lane request was after, kept as a lightweight count rather than a growing list, per your call — full order history remains a backlog item (`07-epic-map.md`'s "Order history & reporting" placeholder), not built here.
- **Data fetching.** `listOrders` (`lib/orderService.ts`) gains two optional filters: `paymentStatus` and `date` (accepts `today`, matched against `confirmedAt`'s calendar day using the server's local time zone — single-tenant MVP, no per-restaurant time zone handling exists yet and none is added here). `GET /api/orders` (`app/api/orders/route.ts`) passes these through as new optional query params, additive to the existing `status` param. The dashboard's poll tick becomes one `Promise.all` of three requests instead of one: `status=pending`, `status=confirmed&paymentStatus=unpaid`, `status=confirmed&paymentStatus=paid&date=today` (the third response is only used for its `length`).

### 2. Card redesign: compact summary, prices, tap-to-open

- **Problem.** Cards currently show table/customer/items/inline buttons but no per-item or total price, and carry the Confirm/Mark Paid buttons directly on the card.
- **Fix.** Each card becomes a compact summary: order number, table number, customer name, item count, **order total** (new — `Σ priceSnapshot × quantity`, computed client-side from the already-fetched `items` array, no API change needed), a status badge, and a relative timestamp ("2 min ago", derived from `createdAt`). All inline action buttons are removed from the card — the entire card is a single tap target that opens a modal. This matches the "click an order and a modal comes out" request directly, and keeps the card visually simple enough to redesign responsively (Decision 4).

### 3. Shared `Modal` component + order-detail actions

- **Problem.** Two near-identical modal implementations already exist (`ConfirmDialog.tsx` in `app/order/[id]/`, `OrderReviewModal.tsx` in `app/order/`) — same backdrop, `role="dialog"`/`aria-modal`, Escape-to-close, `exiting`-flag-driven fade/scale-out with a shared timing constant. A third hand-rolled copy for the dashboard would triplicate chrome that's already been written and tested twice.
- **Fix.** Extract the shared chrome (backdrop, dialog frame, Escape/backdrop-close wiring, enter/exit animation, `prefers-reduced-motion` handling) into one `Modal` component at `app/components/Modal.tsx` — this is the first component genuinely shared across two independent route trees (`app/order` and `app/dashboard`), so it's the first thing to graduate out of "component lives with the feature that uses it" into a small shared `app/components/` folder, rather than picking one feature folder to own it arbitrarily. `ConfirmDialog` and `OrderReviewModal` are refactored to build on top of it rather than duplicating the chrome a third time; their existing props/behavior/CSS classes are preserved so their existing tests keep passing unchanged.
- **Dashboard's `OrderDetailModal`** (new, built on `Modal`): full item list with per-line qty/price/line-total, order total, table/customer/time info, and:
  - Pending order → "Confirm" action + payment toggle (mirrors today's role gating: staff sees "Mark Paid" only, admin also sees "Mark Unpaid" per `INV-9`).
  - Confirmed & Unpaid order → payment toggle only (no re-confirm control — that transition is already terminal in one direction, `INV-5`/state machine unchanged).
  - Confirming inside the modal keeps today's `window.confirm()`-equivalent safety, upgraded to the shared modal's own two-step affordance (a "Confirm order" button inside the already-open detail modal, no separate native `confirm()` popup) — this replaces the current bare `window.confirm()`, which was the only unstyled confirmation surface in the app.

### 4. Motion and responsive layout

- **New Pending order arrives** (detected on a poll tick where a new order id appears): card fades/slides in and a brief highlight pulse plays (~1-2s) before settling to normal card styling, using this codebase's existing animation-token conventions (see `*-arrive`/`*-enter` keyframes already in `app/globals.css`) and respecting `@media (prefers-reduced-motion: reduce)`, consistent with every other animated element in this app.
- **Order confirmed:** its card plays the existing ~200ms exit animation (`CONFIRM_EXIT_MS` convention from `ConfirmDialog`) out of the Pending lane; a new card animates into the Confirmed & Unpaid lane.
- **Marked Paid:** the Confirmed & Unpaid card exits; the "Completed today" counter increments (a small numeric bump animation, not a full re-render flash).
- **Layout.** Each lane uses CSS grid with `auto-fill`/`minmax()` column sizing so cards reflow by available width rather than by fixed device breakpoints — you use a mix of tablet and desktop day to day, so this avoids tuning for one screen size and breaking on the other. `Modal`/`OrderDetailModal` renders as a bottom/full sheet under a narrow-viewport media query and a centered dialog above it, matching common responsive modal practice and this app's existing 44px touch-target discipline.

### 5. Customer confirmation page: live status polling

- **Problem.** `/order/[id]` is a server component with no client-side polling. The customer only learns an order was confirmed by manually reloading, or by attempting an edit and receiving the existing `409`/`CONFLICT_MESSAGE` response. `INV-5`'s enforcement itself is already correct (server rejects mutation, client already omits edit controls once `Confirmed`) — the gap is purely *when the customer's screen reflects it*.
- **Fix.** New endpoint `GET /api/orders/[id]/status`, returning only `{ fulfillmentStatus }`. `TicketCard` (both the Pending and Confirmed views) never renders `paymentStatus` today, so there is nothing on this page for a payment-status change to update — the endpoint and poller only need the one field. It follows the same access model the existing customer-facing `DELETE /api/orders/[id]` and `DELETE /api/orders/[id]/items/[itemId]` routes already use — possession of the order id is the access control; no new auth posture is introduced.
- A new client component, `app/order/[id]/OrderStatusPoller.tsx`, wraps the page's dynamic content, polling this endpoint every 3.5s (matching the dashboard's existing interval/pattern, consistent with ADR-001's rationale that this app doesn't hold persistent connections). On detecting `fulfillmentStatus` flip from `Pending` to `Confirmed`, it swaps the rendered content from the editable `OrderTicket` to the static locked `TicketCard` view in place, reusing the item data the page already has (items are immutable once `Confirmed`, per `INV-5`, so no re-fetch of item data is needed) — no full reload, no dependency on the customer taking an action first. Polling stops the moment `fulfillmentStatus` is no longer `Pending` (`Confirmed` or `Cancelled`) — nothing on this page can change further after that.
- This does not change `OrderTicket.tsx`'s existing 409-handling — that stays as a defense-in-depth backstop for the rare race between a poll tick and an in-flight customer edit.

## Testing

Extends existing co-located tests, per `06b-engineering-decisions.md` §7 (Vitest + RTL):

- **`lib/orderService.test.ts`**: new cases for `listOrders` with `paymentStatus` and `date=today` filters, independently and combined with `status`.
- **`app/api/orders/route.test.ts`**: asserts the new query params are parsed and passed through; invalid `date` value handling (only `today` is accepted at implementation time — anything else is a `400`, per `05-api-conventions.md`'s existing validation pattern).
- **`PendingOrdersDashboard.test.tsx`** (substantially updated): three-lane rendering from the three poll responses; card tap opens `OrderDetailModal` with correct fields/actions per order state; confirming/paying from the modal updates lane placement and the completed-today count; new-order and status-change animations use `vi.advanceTimersByTimeAsync` (this codebase's established fix for async polling `useEffect`s under fake timers, documented in `BUILD_STATUS.md`'s gotchas log) rather than plain `advanceTimersByTime`.
- **New `Modal.test.tsx`**: the extracted shared chrome — Escape/backdrop-close, enter/exit animation flag, `busy` disabling.
- **`ConfirmDialog.test.tsx` / `OrderReviewModal.test.tsx`**: re-run unchanged against the refactored implementation to confirm no behavior regression from the extraction.
- **New `OrderStatusPoller.test.tsx`**: simulates a status transition mid-poll and asserts `OrderTicket` is swapped for the locked `TicketCard` without a page reload; asserts polling stops after a terminal state.
- Pure CSS motion (fade/slide/pulse) has no reliable jsdom assertion — verified manually via `docker compose up --build`, per this repo's established preference for verifying through Docker over host `npm run dev` (`BUILD_STATUS.md` gotchas log).

## Scope boundary — do NOT touch

`confirmOrder`, `setPaymentStatus`, `cancelOrder`, `removeOrderItem` in `lib/orderService.ts` — the state machine and all invariants (`INV-4/5/8/9`) are unchanged; this is a read/presentation-layer pass only. `POST /api/orders` (order creation) and the customer menu/cart flow are untouched. Full order-history/reporting (the `07-epic-map.md` backlog placeholder) is explicitly out of scope — "Completed today" is a same-day count only, not a history feature. Cancelled orders remain out of the dashboard view entirely, per your call — no new lane or count for them in this pass.
