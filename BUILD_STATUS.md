## Gotchas log

- **Vitest 4 mock hoisting.** `vi.mock('mod', () => ({ fn: someOuterConstFn }))` throws `Cannot access '...' before initialization` if `someOuterConstFn` is a plain top-level `const` declared below the `vi.mock` call — `vi.mock` factories are hoisted above all other top-level code. Fix: declare the mock functions via `vi.hoisted(() => ({...}))` and destructure from that. Hit this in `lib/authGuard.test.ts` (Task 7).
- **Prisma 7 requires a driver adapter, not just a `DATABASE_URL`.** `npx prisma init` now defaults to Prisma 7, which removed the implicit query-engine connection — `new PrismaClient()` with no arguments no longer connects to anything. You must install `@prisma/adapter-pg`, construct `new PrismaPg({ connectionString: process.env.DATABASE_URL })`, and pass it as `new PrismaClient({ adapter })`. Also requires a `prisma.config.ts` file (datasource URL and migration/seed config no longer live inline in `schema.prisma`'s `datasource` block). See `lib/prisma.ts`, `prisma.config.ts`, `prisma/seed.ts`.
- **Two Postgres instances can silently fight over port 5432.** If a native Windows Postgres service is already installed, a Docker Postgres container mapped to the same port will also bind successfully (no error), but `localhost:5432` resolves ambiguously (IPv6 `::1` vs IPv4) and traffic may go to either one, causing "my migration ran but the seed I'm querying shows nothing" confusion. Check `netstat -ano | grep :5432` before assuming which instance you're talking to; prefer `127.0.0.1` over `localhost` in `DATABASE_URL` to pin the connection explicitly.
- **`tsc --noEmit` fails with "Property does not exist" / "no exported member" after adding a Prisma model.** Editing `schema.prisma` alone doesn't regenerate `@prisma/client`'s TypeScript types — `npx prisma migrate dev` normally regenerates the client as a side effect, but if the generated client is stale from an earlier session/checkout, `tsc` reports the new model's fields/type as missing even though the schema and migration are correct. Fix: `npx prisma generate`. Hit this in Task 6 of Story 2 (`lib/tableService.ts` briefly appeared broken to `tsc` for reasons unrelated to that task's actual diff).
- **`ConflictError`'s wire code is always `CONFLICT`, never a per-scenario constant — unless you give the scenario its own subclass.** `handleApiError`'s `codeFor()` derives the JSON `error` code from the error *class* name, not the message. This bit for real in Story 23: `addOrderItem`'s sold-out check and `INV-16`'s Paid-order check both threw plain `ConflictError`, so `PendingOrdersDashboard`'s client code — which keyed its "mark this item unavailable" side effect off `err.code === 'CONFLICT'` — fired on *either* conflict, incorrectly greying out a perfectly available item whenever the Paid-lock rejected an add. Live-reproduced via Docker Compose (`ISSUE-28`'s companion finding), not caught by jsdom-mocked tests since the mocks always used the "right" code for whichever scenario each test meant to simulate. Fixed by adding `SoldOutError extends ConflictError` (`lib/errors.ts`) mapped to its own `SOLD_OUT` code in `codeFor()`, checked before the generic `ConflictError` branch since subclass order matters. **The pattern going forward:** when a route's callers need to react to *why* a 409 happened, not just show the message, give that reason its own `ConflictError` subclass — don't have the client guess from message text, and don't reuse the generic `CONFLICT` code for two behaviorally-different scenarios on the same client code path.
- **Testing a polling `useEffect` requires `vi.advanceTimersByTimeAsync`, not `vi.advanceTimersByTime`.** `PendingOrdersDashboard`'s poll function is `async` (it awaits `apiClient.get`); with Vitest 4's plain `advanceTimersByTime`, the fake-timer tick fires but the pending promise inside it never gets a chance to resolve before the assertion runs, so the test either false-passes on stale state or hangs. `advanceTimersByTimeAsync` (wrapped in `await act(async () => { ... })`) flushes the microtask queue after each timer tick, which is what actually lets the second `fetch`/render cycle complete before you assert on it. First use of fake timers in this codebase (Story 7, Task 4) — this pattern should be reused for any future polling/interval component test rather than rediscovered.
- **`userEvent.setup({ advanceTimers })` does not fully fix fake-timer hangs on this repo's Vitest/user-event pairing.** The documented fix for combining `vi.useFakeTimers()` with `userEvent` is `userEvent.setup({ advanceTimers: vi.advanceTimersByTimeAsync })` — but on this repo's exact versions (Vitest 4.1.9 + `@testing-library/user-event` 14.6.1), even with that fix wired correctly, `user.click()` itself hangs and times out under fake timers, before any timer-advance step is reached. The working fix is to use `fireEvent.click(...)` instead of `user.click(...)` for the interaction, wrapped with `await act(async () => { await vi.advanceTimersByTimeAsync(ms) })` for the timer step — the same pattern `PendingOrdersDashboard.test.tsx` already uses. First hit in `app/order/Cart.test.tsx`'s toast auto-dismiss test (Task 2 of the cart-toast/review-modal feature).
- **Plain `.test.ts` files have no `sessionStorage`/`localStorage` — don't "fix" this with a shared polyfill.** `vitest.config.ts` splits tests into two projects: `**/*.test.ts` runs under the `node` environment (no browser globals), `**/*.test.tsx` runs under `jsdom` (real `sessionStorage`). A `.test.ts` file that needs `sessionStorage` (e.g. a plain-TS storage-helper module with no React involved) will fail with it undefined. Do **not** patch this by adding a polyfill to the shared `vitest.setup.ts` — that silently changes behavior for every `.test.ts` file in the repo, and a hand-rolled `Storage` polyfill is easy to get subtly wrong (e.g. `key(index)` ignoring `index`, or `getItem` returning `null` for a stored `""` instead of `""`). Instead add `// @vitest-environment jsdom` as the literal first line of the one test file that needs it, before any imports — this opts just that file into the `jsdom` project's real, spec-correct `sessionStorage` with zero shared-config risk. Hit and reverted in Task 2 of the order-customer-name feature (`app/order/orderNameStorage.test.ts`).
- **Docker Compose is now the actual local dev/test loop, not just an additional isolated option.** The original Docker design (`docs/superpowers/specs/2026-07-06-docker-compose-deployment-design.md`) was written with host `npm run dev` + `.env.local` as the primary workflow and `docker compose up` (app on host `:3001`, Postgres on host `:5433`) as a secondary, isolated way to run the project. In practice, Docker is now how this project is actually run and verified locally. This does **not** change the production deployment plan — `BUILD_STATUS.md`'s Deployment checklist (Vercel + Neon Postgres) is unaffected and still the target for production. When verifying a change works, prefer checking it via `docker compose up --build` over assuming host `npm run dev` was the check.
- **Vercel ignores a `vercel-build` npm script once a framework preset is recognized.** The documented `vercel-build`-script-auto-pickup convention does not apply when the project's Framework Preset is set to "Next.js" (or any other recognized framework) — Vercel runs `npm run build` directly instead and never looks for `vercel-build`. The build then fails at `next build` with `Module '"@prisma/client"' has no exported member '...'`, because `prisma generate` (part of the `vercel-build` script, not `build`) never ran. Fix: explicitly override the Build Command in Vercel's Project Settings → Build and Development Settings to `npm run vercel-build`. Discovered via a real failed deploy during the first production rollout; see `docs/superpowers/plans/2026-07-08-production-deployment.md` Task 4 Step 3. **Update:** that dashboard override only lives in Vercel project state, not in git — if the Vercel project were ever recreated, the override would be lost with nothing in the repo to catch it or explain the fix. A root-level `vercel.json` (`{ "buildCommand": "npm run vercel-build" }`) was added specifically to make this fix survive project recreation — Vercel honors `vercel.json`'s `buildCommand` regardless of the recognized framework preset. The dashboard override is no longer required (it's now redundant/backup) but is harmless to leave set.
- **Vercel Hobby's custom-domain "Add Domain" dialog only offers Production/Preview, not a per-branch picker — but assigning it while viewing a specific branch's deployment still pins it to that branch.** There's no explicit "assign to Git branch" option shown on Hobby, which looks like it means a custom `*.vercel.app` domain assigned to "Preview" would alias whichever branch deployed most recently, project-wide. Empirically this isn't what happens: assigning `automatic-fortnight-dev.vercel.app` from the `dev` branch's deployment page and `automatic-fortnight-preprod.vercel.app` from `preprod`'s kept them correctly isolated (verified by pushing a marker file to `dev` only and confirming it 404s on the `preprod` URL). Also note: all non-Production deployments are behind Vercel's SSO wall (Deployment Protection) by default — disable it in Settings → Deployment Protection if the URL needs to be viewable without a Vercel login. - **Concurrent deploys against the shared Neon database can fail with a Postgres advisory-lock timeout, not a real bug.** `prisma migrate deploy` takes an advisory lock (`pg_advisory_lock`) for the duration of the migration step. Since `dev`, `preprod`, and every feature-branch preview share the single production database, pushing to two branches close together can trigger two builds whose `migrate deploy` steps overlap — the second one times out after 10s with `Error: P1002 ... Timed out trying to acquire a postgres advisory lock`, and the whole `vercel-build` fails. Fix: just redeploy once the colliding build finishes: this only fails from a genuine collision, not app or migration state corruption.
- **A `date=today` filter needs its own `confirmedAt`/`paidAt`-equivalent field — don't optimistically bump a count scoped by one field based on a different action.** `listOrders({ date: 'today' })` (`lib/orderService.ts`) scopes "today" by `confirmedAt`. The dashboard's "Completed today" count is that same query's result length. An early version of the staff-dashboard redesign optimistically incremented the local count the instant a Confirmed & Unpaid order was marked Paid — but that lane is deliberately *not* date-scoped (an order confirmed yesterday can still be sitting there unpaid), so paying off a carried-over order bumped the count, then the very next poll (≤3.5s later) correctly excluded it again (its `confirmedAt` wasn't today) and the number visibly dropped back down. Fix: only optimistically bump a count when the action you just took is *itself* what makes the record match the filter (e.g. confirming an already-Paid order — `confirmedAt` becomes "now" as part of that exact API call, so it's safe); for any other action, let the next poll be the sole source of truth rather than guessing client-side. Caught in the final whole-branch review of the dashboard redesign, not by any per-task review — this class of bug only shows up when the filter's field and the action's effect are viewed together.
- **A shared Next.js layout doesn't automatically refetch its server data on a client-side `router.push()` — you have to force it with `router.refresh()`.** `app/layout.tsx` (the root layout, common ancestor of every route) calls `peekSession()` to decide whether to render the global `StaffBar`. `app/login/page.tsx` originally did `await apiClient.post('/api/auth/login', ...); router.push('/dashboard')` with no refresh. Since the root layout was last rendered while still on `/login` (session was `null` then), and the App Router reuses a shared layout's prior render across a `push()` transition that doesn't require the layout itself to change, the bar stayed invisible immediately after logging in — only a hard reload picked up the new session cookie. Fix: add `router.refresh()` right after the `push()`, which forces the whole route tree (including shared layouts) to refetch server data. Caught via a Docker/Playwright smoke test, not the code review — this class of bug only shows up when you actually click through the flow in a browser, since unit tests mock `useRouter` and never notice a missing `refresh()` call. Any future session-dependent logic placed in a shared layout needs the same treatment on whatever page transitions into it after a session change (see `StaffBar`'s own logout handler, which already does `push('/login'); refresh()` for the same reason).
- **`resolveBranchId(session)` silently defaults an admin to the *first* branch — any admin-reachable page that resolves a branch must be wired into the header's `?branch=` propagation, not just added to the route tree.** When the header branch picker was introduced (2026-07-11 redesign), `StaffBar.BRANCH_AWARE_PATHS` was the list of pages that (a) get the selected branch appended to their nav `href` and (b) re-navigate when the branch is switched — but it only covered `/dashboard`, `/admin/menu-items`, `/admin/tables`. The staff-assisted `/order/new` flow (linked from the dashboard's "+ New order" button, not a `StaffBar` nav link) was left calling `resolveBranchId(session)` with no `requestedBranchId`; for an admin (no `session.branchId`) that falls through to `listBranches()[0]`, so every admin-placed order silently landed in the first branch regardless of the selected one (ISSUE-24). Fix pattern: the linking page passes its own selected `?branch=` into the target href, and the target page reads `searchParams.branch` and resolves against it — with an explicit branch-chooser step when the admin has "All branches"/nothing selected, rather than defaulting to one branch. Staff (branch-scoped `session.branchId`) were never affected; the bug is admin-only, which is why the branch-scoped-staff tests didn't catch it. When adding any new page that turns a selected branch into a concrete action (placing an order, editing branch-scoped data), check whether the admin's selection actually reaches it end-to-end — don't assume `resolveBranchId` "just knows."
- **`visibility: hidden` in a `@media print` isolation block doesn't clear a painted ancestor background — you still need an explicit `background: #fff` on the surviving element(s).** The classic "print only this one element" pattern (`body * { visibility: hidden } .receipt, .receipt * { visibility: visible }`) hides all *content*, but `body`'s own `background` (this app's dark espresso theme) is still painted underneath, since `body` itself is never targeted by the `visibility: hidden` selector. First build (Story 22, receipt printing) shipped with unreadable black-on-dark-background print output — invisible in jsdom (doesn't render `@media print` at all) and easy to miss by eye since the on-screen modal never shows this state. Caught via a real Playwright smoke test using `page.emulateMedia({ media: 'print' })` + a full-page screenshot, which is the only way to actually see print-stylesheet output without a physical printer. Fix: explicitly set `background: #fff` on `html`, `body`, and the printed element itself inside the `@media print` block — don't rely on the absence of a background counting as white. Reuse this pattern (and the emulate-print-media-then-screenshot verification technique) for any future print-only UI.
- **`visibility: hidden` also reserves full layout height, and `@page { size: <width> auto }` (flexible height) isn't reliably honored outside an interactive print dialog — both combine into an oversized printout.** Follow-up on Story 22 (receipt printing) after real user testing: the printed/PDF output was a full Letter-sized page (`/MediaBox [0 0 612 792]`, confirmed by grepping a Playwright-generated PDF) with the tiny receipt tucked in the corner, because (1) `visibility: hidden` keeps the whole hidden dashboard's height reserved in the print layout even though nothing renders, and (2) Chromium's headless PDF export (`page.pdf({ preferCSSPageSize: true })`, and likely some browsers'/printers' "print to PDF" drivers too) silently ignores a flexible `auto` page height and falls back to the default page size instead of sizing to content. Fix, two parts: (a) the receipt now portals (`createPortal`) to a dedicated `<div id="receipt-print-root">` appended directly to `document.body`, so the print stylesheet can `display: none` (zero layout height, not just invisible) everything else via `body > *:not(#receipt-print-root)`; (b) a `beforeprint` listener measures the receipt's actual rendered height and injects a concrete two-dimension `@page { size: 80mm <Nmm>; margin: 0 }` rule right before printing, rather than relying on `auto` — see `useReceiptPageSize` in `app/dashboard/OrderDetailModal.tsx`. Verified by grepping the regenerated PDF's `/MediaBox` (80mm × content height, not 612×792). Reuse this pattern (portal-per-print-target + measure-then-inject-`@page`-size) for any future print-only UI in this app — `auto` page sizing is not a safe default to rely on. **Second follow-up, same day:** that fix itself shipped with a bug — the portal was hidden via `display: none` by default (shown only inside `@media print`), so measuring its height via `getBoundingClientRect()` on `beforeprint` raced against whether print styles had actually applied yet; when it hadn't, the element read as 0-height, producing a ~5mm-tall page that paginated the receipt one line per sheet (confirmed live: a 20-page PDF, one fragment of text per page). Root cause: `display: none` isn't just invisible, it's *unmeasurable* — nothing to lay out. Fixed by never `display: none`-ing the portal at all; it's positioned off-screen instead (`position: fixed; left: -9999px`), which keeps it fully laid out (and therefore reliably measurable) at all times regardless of print-style timing, and `@media print` merely repositions it on-page (`position: static`) rather than toggling its display. General lesson: never measure an element's dimensions while it (or an ancestor) is `display: none` — use off-screen positioning instead when a "hidden but must stay measurable" element is needed.
- **`align-items: stretch` in a flex row stretches every sibling to the *tallest content*, not to any bounded container height — a real trap for two panes with very different content lengths.** Story 23's two-pane `OrderDetailModal` (menu picker left, order ticket right) gave `.order-detail-modal__panes` `align-items: stretch` with neither pane assigned its own `overflow-y`/height. With a 25-item picker (~1700px of content) next to a 1-3-line ticket, the ticket pane stretched to match the picker's full height rather than staying within the modal's intended `max-height: 85vh` — pushing the ticket's pinned total and settle buttons (`Confirm`/`Mark Paid`/etc.) hundreds of pixels below the visible viewport, requiring the user to scroll almost the entire picker to reach them (`ISSUE-27`). Invisible in jsdom, which has no real layout engine — only surfaced via a live Docker Compose + Playwright pass with real seeded menu data (25 items), not the 1-3-item fixtures the unit tests use. Fix: give each pane its own `min-height: 0` + `overflow-y: auto` so it scrolls independently within the flex container's own bounded height, instead of letting `stretch` size one pane off the other's content. General lesson: any multi-pane flex/grid layout where panes can have very different content lengths needs this pattern — don't assume `stretch` respects an ancestor's `max-height`.
- **A `POST`/`PATCH` handler's response type parameter (`apiClient.post<T>(...)`) is an unchecked assertion, not a runtime guarantee — `tsc` will not catch a mismatch, and an over-generous test mock will hide it too.** Story 23's `handleAddItem`/`handleAdjustQuantity` called `apiClient.post<DashboardOrder>('/api/orders/:id/items', ...)` and spliced the "response" wholesale into state. The actual server response is `lib/orderService`'s `OrderWithItems` (`items` + base `Order` columns only — `include: { items: true }`, no `orderingPoint`/`branch`), not the full `DashboardOrder` the type parameter claimed. `tsc` had nothing to check against (the generic just told TypeScript to trust the assertion), and the unit tests' mocked responses always spread the full test fixture (`{ ...orderA, items: [...] }`), which happened to include the "missing" fields and so never exercised the real shape. Splicing the narrow response into state wiped `order.orderingPoint`, and `OrderDetailModal` crashed on `order.orderingPoint.label` the instant a mutation succeeded — closing the dialog outright (`ISSUE-28`). Only caught via a live Docker Compose + Playwright pass against the real API. Fix, two parts: (a) merge only the field that actually changed into the existing order snapshot (`{ ...order, items: updated.items }`) instead of trusting the response's shape for fields it doesn't carry — the safer pattern `handleSetPaymentStatus` already used; (b) narrow the mutation endpoints' test mocks to return only what the real API returns (`{ id, items }`, not the full fixture), so a regression back to `spliceOrder(response)` fails the unit tests too, not just a live browser. General lesson: when a handler only needs one or two fields off a mutation response, type and mock exactly those fields — typing (and mocking) the full domain object is an invitation for this class of bug to hide until a live smoke test finds it.
- **A shared CSS class used with different children needs the alignment fix applied at every call site, not just the first one found.** `ISSUE-30` and `ISSUE-31` (Story 23) were the *identical* bug — flex `justify-content: space-between` sizing a right-hand group off its own content instead of a fixed column, so the group's contents landed at a different x depending on whether an optional child (a remove button, a count badge) was present — but they lived in two different components (`OrderTicketPane`'s ticket lines and `MenuItemPicker`'s picker rows) sharing no common CSS class. Fixing one didn't fix the other; both needed the same flex-to-grid conversion applied independently. When a layout bug's *root cause* is a general pattern ("flex sizes a group off its own content, not a shared column"), grep for other places the same pattern appears before calling the fix done — don't assume fixing the one reported instance is complete.
- **A relation fetched via Prisma's `include` with no explicit `orderBy` has unguaranteed row order — and a shared `DateTime @default(now())` does NOT fix it if the rows are inserted in one batch.** Story 23's `ISSUE-32`: `OrderItem` had no field reflecting insertion order (`id` is a random UUID, not time-ordered), so every `include: { items: true }` in `lib/orderService.ts` (12 call sites) relied on Prisma's unordered default — empirically stable until an `UPDATE` touched a row's physical position, after which the touched item visibly jumped to the end of the list on every quantity/add/remove mutation. **The first fix attempt looked right and wasn't**: adding `createdAt DateTime @default(now())` and ordering by it asc still reproduced the bug identically, because `createOrder` inserts every line item in *one batch* inside a single `prisma.order.create()` call — every row in that batch gets the exact same millisecond timestamp, so `ORDER BY createdAt ASC` has nothing to break the tie with and falls back to the same unordered behavior. The actual fix needed a value guaranteed to differ per row even within one transaction/batch insert: `sequence Int @default(autoincrement())`, a real DB sequence, not a shared clock reading. General lesson: for "preserve insertion order" semantics on a to-many relation whose children are ever created in a batch, reach for `autoincrement()`/a sequence, not a timestamp default — timestamps have finite precision and a batch insert can produce identical values for every row in the batch.
- **The Vercel↔GitHub connection can silently drift out of sync — Vercel's own dashboard says "Connected," but nothing actually triggers.** Discovered 2026-07-23/24: pushes to `dev` stopped triggering Vercel builds with zero error anywhere — the push succeeded, `origin/dev` updated correctly, but no deploy queued. **The repo's GitHub Settings → Webhooks page is not useful for diagnosing this** — it only lists classic per-repo webhooks manually added through that exact UI; Vercel's integration is a GitHub App, whose event delivery is configured at the App level and never appears there, working or not. Verified working again by pushing to both `dev` and `preprod` and confirming builds queued in Vercel.
